# GOAT Known Limitations (Public Alpha)

GOAT is in **Active Alpha**. While many workflows function as intended, you should be aware of the following limitations before running real-world tasks:

## Core & Storage
* **Data Storage:** Public alpha is meant for local-first experimentation. All data lives in the local XDG path (`~/.local/share/goat`). There is currently no cloud sync or backup beyond manual export.
* **Brain Indexing:** Semantic search and Brain context ingestions are functional but may not scale efficiently for massive repositories (>50,000 files) yet.
* **Model Providers:** External API providers (OpenAI, Groq, OpenRouter) require explicit manual configuration in `goat.toml` or via `goat setup`. They are disabled by default. 

## UI Interfaces
* **Dashboard Reconnection:** If the Daemon crashes or is restarted, the dashboard might require a hard page refresh to restore the WebSocket / state.
* **Desktop App:** The Tauri-based desktop app scaffold is experimental and lacks full native integrations.

## Agent Workflows
* **Browser Workflows:** Visual/Browser workflows depend entirely on the presence of local browser engines (Playwright/Chrome). They may fail silently or timeout if a headless browser cannot be spawned.
* **Builder:** The Builder cannot auto-deploy or auto-commit. This is by design, but can increase friction for experienced users wanting fully autonomous swarms.
* **Operator:** Deployment plans generated by Operator cannot be executed autonomously. They are currently structured as read-only runbooks.
* **Socializer:** Will draft posts but lacks the Oauth plumbing to automatically post to Twitter/LinkedIn.

*We are continuously improving these features. See our Issue Tracker for up-to-date bugs.*
