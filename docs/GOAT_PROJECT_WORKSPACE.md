# GOAT Project Workspace

## Concept
Currently, operations in GOAT (reports, timeline events, agent executions) are highly task-focused but disconnected from a larger unifying structure. The Project Workspace solves this by grouping related artifacts, agents, and history under a single context.

## Model
The core entity is the `GoatProject`.
If a project profile system exists within GOAT, we will upgrade it. Otherwise, we define it as:

```rust
pub struct GoatProject {
    pub id: String, // GoatProjectId
    pub goal: String, // GoatProjectGoal
    pub status: String, // GoatProjectStatus
    pub agent_refs: Vec<String>, // GoatProjectAgentRef
    pub artifact_refs: Vec<String>, // GoatProjectArtifactRef
    pub context_refs: Vec<String>, // GoatProjectContextRef
    pub next_action: Option<String>, // GoatProjectNextAction
}
```

## Connections
A project binds together:
- **Agents**: The recommended or active team (e.g., Builder + Designer).
- **Reports**: Markdown outputs generated by agents.
- **Timeline Events**: Chronological history of the project's lifespan.
- **Runtime Jobs**: Background tasks currently running for this project.
- **Brain Docs**: Memory and semantic context ingested specifically for this project's boundaries.
- **Approvals**: Risk gates triggered during this project's execution.

## Usage
When the user visits Mission Control, they select an Active Project (or create one by submitting a new Goal). Once selected, the Mission Control view filters the Unified Activity Feed, Artifacts, and Brain Context to show only items relevant to that `GoatProject`.
