# Release Process Guide

This document outlines the procedure for publishing a new version of GOAT.

## Release Checklist

Before tagging a release, ensure the following steps are completed:

1. **Test Suite:** Ensure `cargo test` passes 100%.
2. **Dashboard Build:** Ensure the Next.js dashboard compiles cleanly via `npm run build`.
3. **Docs Update:** Verify `CHANGELOG.md` is updated with the latest changes and features are correctly marked in `GOAT_FEATURE_MATRIX.md`.
4. **Version Bump:** Update the version in `Cargo.toml`, `apps/dashboard/package.json`, and any other hardcoded areas (e.g., about page).
5. **Security Audit:** Ensure no secrets or API keys have been committed.

## Tagging & Publishing

We use semantic versioning with alpha/beta tags during early development.

```bash
git tag v0.1.0-alpha.1
git push origin v0.1.0-alpha.1
```

## Binary Packaging Plan

* **Linux:** Provide pre-compiled binaries via GitHub Releases.
* **macOS:** Explore Homebrew formula publishing once the CLI is stable.
* **Windows:** Provide `.exe` via GitHub Releases.
* **Desktop:** Distribute AppImage, `.dmg`, and `.msi` generated by Tauri.

## Rollback Plan

If a release is determined to have critical bugs:
1. Mark the GitHub Release as "Pre-release" or remove it if severe.
2. Publish an immediate patch release (e.g., `v0.1.0-alpha.2`) with fixes.
3. Update users via the Changelog.
